# Harness Engineering：为什么 AI Agent 真正的瓶颈，不再只是模型本身

最近在 AI 圈里，`Harness Engineering` 这个词越来越火。很多人在做 Agent、工作流或者 AI 应用落地时，都会遇到一个很现实的问题：

同样的模型，为什么别人做出来的 Agent 能稳定运行很久、成功率很高，而自己做出来的系统却总是忽好忽坏？

很多人第一反应是：

- 模型不够强
- Prompt 没写好
- RAG 没调明白

这些当然都会影响效果，但越来越多团队发现，真正决定系统能不能稳定运行的，往往不是模型本身，而是模型外面那套“运行系统”。

这套东西，现在有了一个更统一的名字：`Harness`。

## 一、AI 工程的三次重心迁移

过去两年，AI 工程经历了三次非常明显的重心变化：

- `Prompt Engineering`
- `Context Engineering`
- `Harness Engineering`

这并不只是几个新名词的流行，而是 AI 系统发展过程中，三个不断向外扩展的问题层次。

### 1. Prompt Engineering：让模型听懂你在说什么

最早大家关注的是提示词。因为同一个模型，换一种说法，效果可能天差地别。

Prompt Engineering 的核心，不是“命令模型”，而是通过角色设定、格式约束、示例、风格说明等方式，去塑造模型当前的概率空间，让它更容易朝着你期待的方向输出。

它解决的是一个问题：

`你有没有把任务讲清楚。`

但 Prompt 的边界也很明显。它擅长表达、约束、激发模型已有能力，却不擅长弥补缺失知识、管理复杂状态和处理长链路任务。

### 2. Context Engineering：让模型拿到正确的信息

当 AI 应用从单轮问答走向 Agent 之后，问题开始变化。模型不再只是回答一句话，而是要完成多步骤任务，要调用工具、读取外部资料、利用中间结果持续推进。

这时候，系统面对的问题不再只是“回答对不对”，而是：

`模型有没有在合适的时机拿到正确的信息。`

所以 Context Engineering 的核心变成了：

`系统必须把正确的信息，在正确的时机送进模型。`

这里的 context 不是几段背景材料那么简单，而是所有会影响模型当前决策的信息总和，包括：

- 用户输入
- 历史对话
- 检索结果
- 工具返回
- 当前任务状态
- 中间产物
- 系统规则
- 安全约束
- 其他 Agent 的结构化输出

从这个角度看，Prompt 其实只是 Context 的一个子集。

### 3. Harness Engineering：让模型持续做对

即使 Prompt 写对了，Context 也喂对了，模型依然可能在真实执行中跑偏。

比如：

- 计划看起来很好，但执行偏了
- 工具调用了，但结果理解错了
- 长链路任务中逐步偏航，系统却没有发现
- 出错之后没有恢复机制，只能从头再来

这时候，问题已经不只是输入层面的问题，而是运行过程的问题。

于是第三阶段出现了：`Harness Engineering`。

它关注的核心不是“模型更会想”，而是：

- 怎么让模型别跑偏
- 怎么让系统跑得稳
- 出错以后怎么拉回来
- 最终怎么验收结果

一句话总结：

`Prompt 解决表达问题，Context 解决信息问题，Harness 解决运行问题。`

## 二、什么是 Harness

`Harness` 这个词原本的意思是缰绳、马具、约束装置。

放到 AI 系统里，它强调的是：当模型从“回答问题”走向“执行任务”之后，系统不仅要提供信息，还要能够驾驭整个过程。

可以用一个非常形象的类比来理解：

假设你要派一个新人去完成一次重要客户拜访。

- `Prompt Engineering`：告诉他话要怎么说，流程是什么
- `Context Engineering`：把客户背景、历史记录、报价资料、会议目标准备齐
- `Harness Engineering`：给他 checklist，要求关键节点汇报，会后核对纪要，发现偏差及时纠正，并按标准验收结果

所以三者不是替代关系，而是包含关系：

- Prompt 是对指令的工程化
- Context 是对输入环境的工程化
- Harness 是对整个运行系统的工程化

## 三、Agent = Model + Harness

视频里引用了一个很经典的定义：

`Agent = Model + Harness`

反过来说：

`Harness = Agent - Model`

这句话的意思很直接：

在一个 Agent 系统里，除了模型本身之外，几乎所有决定它能不能稳定交付的东西，都可以算进 Harness。

也就是说，真正影响 Agent 落地效果的，不只是模型能力，而是模型外面那套系统设计。

## 四、一个成熟的 Harness 包括哪六层

作者把成熟的 Harness Engineering 拆成了六层，这个拆法非常适合拿来做理解框架。

### 1. 上下文层

目标是让模型在正确的信息边界内思考。

核心包括：

- 明确角色、任务和成功标准
- 裁剪和选择相关信息
- 对信息进行结构化分层组织

重点不是“给得越多越好”，而是“给得越准越好”。

### 2. 工具层

模型如果没有工具，本质上仍然只是一个文本生成器。接上工具之后，它才能真正接触世界，比如：

- 搜网页
- 读文档
- 写代码
- 调 API
- 查数据库

但工具不是挂上去就完了，系统还要解决：

- 给哪些工具
- 什么情况下调用
- 工具结果如何整理后再回灌给模型

### 3. 执行编排层

解决的是“下一步该做什么”。

很多 Agent 不是单步能力不行，而是不会把多个步骤串起来。它可能会搜索、会总结、会写代码，但整个过程想到哪做到哪，最后只交出一堆半成品。

所以 Harness 要提供执行轨道，例如：

- 理解目标
- 判断信息是否足够
- 不够就补
- 基于结果继续分析
- 生成输出
- 做校验
- 不符合要求就修正或重试

### 4. 记忆与状态层

没有状态管理的 Agent，每一轮都像失忆。

因此系统至少要区分三类东西：

- 当前任务状态
- 会话中的中间结果
- 长期记忆与用户偏好

如果这些混在一起，系统会越来越乱。

### 5. 评估与观测层

很多团队容易忽视这一层。系统不是“能生成”就够了，而是要知道自己“生成得对不对”。

这层通常包括：

- 输出验收
- 环境验证
- 自动测试
- 日志与指标
- 错误归因

也就是说，系统不仅要会做，还要知道自己做得怎么样。

### 6. 约束、校验与失败恢复层

这是决定系统能不能上线的关键层。

因为在真实环境里，失败不是例外，而是常态。可能是搜索失败、API 超时、文档异常，也可能是模型理解错任务。

一个成熟的 Harness 必须具备：

- 约束：哪些能做，哪些不能做
- 校验：输出前后如何检查
- 恢复：失败后如何重试、回滚、从稳定状态继续

## 五、为什么 Harness Engineering 会突然爆火

这个概念之所以火，不只是因为大家在讨论方法论，而是因为越来越多一线公司已经把它变成真实工程实践。

### 1. Anthropic：上下文重置与生产验收分离

Anthropic 在长链路自主任务中发现两个典型问题：

- 上下文越跑越满，模型开始丢细节、丢重点，甚至急于草草收尾
- 模型自己干活、自己评分时，往往会高估自己的结果

为了解决第一个问题，他们提出了 `context reset`。不是继续在原上下文里压缩，而是直接把任务交接给一个全新的 Agent，让它在干净上下文中继续工作。

这本质上就是一种非常典型的 Harness 设计。

为了解决第二个问题，他们把生产者和验收者拆开：

- `planner` 负责规划规格
- `generator` 负责执行实现
- `evaluator` 负责像 QA 一样做真实验证

重点是验收必须独立，系统才能形成有效闭环。

### 2. OpenAI：工程师的职责从写代码转向设计环境

视频中提到，OpenAI 的一个重要思路是：

在 Agent 时代，工程师不一定亲自写每一行代码，而是更多负责设计环境。

具体包括三件事：

- 把产品目标拆成 Agent 可理解的小任务
- Agent 出问题时，不是让它“更努力”，而是追问系统缺了什么结构性能力
- 建立反馈链路，让 Agent 能看到自己工作的真实结果

这是一种非常典型的 Harness 思维：

`当 Agent 表现不佳时，优先改环境，而不是只怪模型。`

OpenAI 还强调“渐进式披露”。不是把所有规则、文档、约束一次性塞给模型，而是先给索引，需要时再动态加载细节。

另外，他们还让 Agent 不只是“写代码”，而是“看见应用”：

- 可以接浏览器
- 可以截图
- 可以操作页面
- 可以看日志和监控
- 可以在隔离环境中独立运行和验证

这就让 Agent 从“会生成代码”变成“会检查结果、发现问题、继续修复”。

## 六、Harness Engineering 的真正意义

这段内容最值得记住的一点是：

AI 落地的核心挑战，正在从“让模型看起来更聪明”，转向“让模型在真实世界里稳定工作”。

所以：

- 当任务还是简单单轮生成时，Prompt 很重要
- 当任务开始依赖外部知识和动态信息时，Context 很关键
- 当模型进入长链路、可执行、低容错的真实场景时，Harness 几乎不可避免

这也是为什么同样的模型，在不同产品中表现差距会非常大。

真正决定上限的，也许是模型。  
但真正决定能不能落地、能不能稳定交付的，往往是 Harness。

## 七、我的总结

如果用一句最精炼的话来概括：

`Harness Engineering，就是把模型从“会回答”变成“会稳定交付”的那套系统工程。`

它不是在替代 Prompt，也不是在替代 Context，而是在更大的系统边界上，把二者都纳入进来，最终解决 Agent 落地过程中最难的那部分问题：

- 如何编排
- 如何观测
- 如何纠偏
- 如何验收
- 如何恢复

未来 AI 应用真正的竞争力，可能不只是“你用了哪个模型”，而是：

`你有没有构建出一套能让模型长期稳定工作的 Harness。`
